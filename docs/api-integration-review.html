系統架構狀態

LoveFu AI 客服系統具備完整的 Shopline 與 WMS 暢流 API 整合路徑，支援 Mock 與生產環境雙模式。兩套 API 均通過統一的安全防護層（cs-guard）進行 whitelist、blacklist 與 PII 遮蔽。

防護入口點

防護函式 協議 HTTP 客戶端 用途
shopline_safe_get() 非同步 httpx.AsyncClient Shopline REST API 查詢
wms_safe_get() 同步 httpx.Client WMS 暢流 API 查詢

防護機制： 兩個防護函式均實作 Endpoint whitelist（僅允許預核准端點）、IP blacklist、以及 PII 遮蔽（客戶電話/Email/姓名）。

⚠️ 發現的問題

🔴 Shopline 商品 API 未接

問題：/products.json 在 guard 白名單中，但 cs-shopline 沒有 query_products.py 實作。PRODUCT 意圖目前只靠知識庫 markdown 檔，無法即時取得 Shopline 上架的商品價格與規格。

影響： 顧客問「山丘床墊多少錢」時，AI 只能回知識庫的預設價格，無法反映最新檔期折扣或庫存狀態，導致銷售機會流失。
修復建議： 新增 cs-shopline/query_products.py，在 PRODUCT 意圖時 parallel fetch Shopline 產品清單，並與知識庫交叉驗證。
🔴 Shopline 庫存 API 未接

問題：/inventory_levels.json 在 guard 白名單中，但未實作相應查詢函式。STOCK 意圖目前只走 WMS stock_query.php，但 Shopline 本身也有庫存欄位，兩邊資料可能不同步。

影響： 顧客問「有沒有貨」時，WMS 庫存可能與 Shopline 前台顯示不一致，尤其在剛下單或取消時易有延遲。
修復建議： STOCK 意圖同時查 Shopline inventory_levels + WMS stock，取較小值（保守策略，優先扣 Shopline）。
🟡 訂單查詢缺乏 LINE UID 映射

問題：query_orders_by_buyer 需要 member_id，但顧客從 LINE 進線時，Make.com 不一定帶 member_id。目前 fallback 到手機/Email 模糊搜尋，但成功率取決於顧客是否主動提供。

影響： 第一次對話的顧客，如果沒報手機號或訂單號，系統就查不到訂單歷史。
修復建議： 新增 Shopline customers/search.json 用 LINE UID 反查 member_id 的 mapping flow，或從 Make.com webhook 帶入 Shopline customer ID。
🟡 Token 無自動續期機制

問題：SHOPLINE_ACCESS_TOKEN 是靜態環境變數，Shopline OAuth token 有效期通常 30 天。超過有效期後，API 呼叫會返回 401 Unauthorized，但系統未設置自動偵測與更新。

影響： 三十天后無人工介入情況下，所有 Shopline 查詢會 silent fail，顧客無法查訂單。
修復建議： 實作 OAuth refresh token flow（Shopline 支援），或至少加入 token 過期偵測 + 告警機制。
🟢 已修復：async 錯誤

v2.2 更新修正了 fulfillment 查詢的 async/sync 混用問題。之前用 run_in_executor 包裝 async 的 query_fulfillment_by_order，會在 thread pool 中無法正確 await。已改為直接 await 非同步版本。

⚠️ 發現的問題

🔴 WMS 同步呼叫阻塞 Event Loop

問題：wms_safe_get() 使用同步 httpx.Client，但 cs-brain 執行在非同步 FastAPI 中。v2.2 版本在 _fetch_context 層加了 run_in_executor 包裝，但 wms_client.py 內部函式（_fetch_orders_uncached、_fetch_cargo_uncached 等）仍為同步且直接從同步 cache 層呼叫。當 cache miss 時，這些同步函式會直接 block 整個 event loop。

影響： 當 WMS API 回應慢（>2 秒）時，會阻塞整個 event loop，影響其他顧客的回應速度。高流量時可能造成 FastAPI 連接排隊。
修復建議： (A) 改 wms_safe_get 為 async httpx.AsyncClient，或 (B) 在 wms_client 層面全面用 run_in_executor 包裝同步函式。建議用 (A) 以保持架構一致性。
🟡 WMS Token 單例非 Thread-Safe

問題：wms_auth._token_cache 是 plain dict，多 worker / 多個 asyncio task 並行時可能 race condition。JWT token refresh 邏輯無 lock 保護。

影響： 低機率情況下，多個 task 同時偵測到 token 過期，同時發起 refresh，導致 token 衝突或不必要的重複請求。
修復建議： 在 wms_auth.py 加入 threading.Lock 或改用 asyncio.Lock（如果改為 async）保護 token refresh。
🟡 WMS PII 解密 Key 管理

問題：WMS_PII_AES_KEY 直接放在環境變數。如果 key 錯誤或未設定，pii_decrypt._aes_decrypt 會默默回傳密文原文，下游 format 函式會顯示 base64 密文給 LLM。

影響： LLM 看到 base64 加密字串，會直接轉述給顧客，洩露加密資料或回覆包含無意義亂碼。
修復建議： 應用啟動時驗證 AES key 是否正確（解一筆已知密文比對），若失敗則 fail-fast。
🟡 wms_client.py Import 路徑不一致

問題：cs-logistics/wms_client.py 第 16-17 行嘗試 from lovefu_cs_guard.api_guard import wms_safe_get（缺少 .scripts），正確路徑應為 lovefu_cs_guard.scripts.api_guard。

影響： 生產環境時可能 ImportError，導致 wms_client 走 fallback stub，返回 mock 資料。
修復建議： 改為正確的 from lovefu_cs_guard.scripts.api_guard import wms_safe_get。
🟢 快取策略合理

終態物流（24h）、中間態訂單（5min）、庫存（2min）、門市（60min）的快取時間符合 WMS 資料更新頻率，無需調整。
