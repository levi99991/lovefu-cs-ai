2. cs-brain 生命週期

完整的 8 步消息處理流程

流程總覽

Step -1: 冪等性檢查 (Idempotency Check)
使用消息 ID 檢查是否已處理過相同消息
Step -0.5: 消息去重 (Message Debounce)
1.5 秒時間窗口，Leader/Follower 機制防止重複
Step 0: 多渠道共存檢查 (Omnichat Coexist Check)
如果客戶已在另一個渠道聊天，靜音當前消息
Step 1: 升級關鍵詞攔截 (Escalation Keyword Intercept)
立即檢測 P0 信號 (例: 投訴、緊急、人工)
Step 2: 加載記憶 (Load Memory)
從 Redis/數據庫 檢索用戶對話歷史和偏好
Step 3: 意圖分類 (Intent Classification)
兩階段: 快速路徑 (0.75 閾值) → LLM 回退
Step 3.5: 生成立即確認 (Generate Instant Ack)
根據意圖生成快速確認文本，立即返回客戶
Step 4: 並行抓取上下文 (Parallel Context Fetch)
asyncio.gather 同時獲取訂單、庫存、門店信息
Step 5: 組裝提示詞 + LLM 生成 (Assemble Prompt + LLM Generate)
雙提供商回退: OpenAI → Anthropic
Step 6: 保存記憶 (Save Memory)
更新對話歷史、用戶偏好、交互記錄
Step 7: 轉接檢查 (Handoff Check)
自動檢測是否需要轉接人工 (情感、高價值、低置信度)

詳細步驟說明

Step -1: 冪等性檢查

  • 使用消息 ID 和時間戳檢查
  • 避免重複處理相同消息
  • 實現: Redis 鍵過期時間 5 分鐘

Step -0.5: 消息去重

  • 時間窗口: 1.5 秒
  • Leader/Follower 機制: 第一條為 leader，後續為 follower
  • Follower 消息在窗口內自動丟棄
  • 防止網絡抖動導致的重複

Step 0: 多渠道共存檢查

  • 檢查用戶是否在 LINE / Omnichat / 門店等渠道同時活動
  • 優先級規則: LINE > Omnichat > 其他
  • 低優先級渠道收到消息時標記為 muted

Step 1: 升級關鍵詞攔截

  • 檢查消息內容是否含有 P0 關鍵詞
  • 關鍵詞示例: 投訴、緊急、人工客服、問題、建議
  • 如果檢測到 P0，立即觸發轉接流程
  • 不進行 LLM 調用，直接返回轉接確認

Step 2: 加載記憶

  • 查詢 MemoryStore / D1 獲取用戶歷史
  • 加載項目: 對話歷史、購買記錄、偏好、上次交互時間
  • 超時: 2 秒，超時則返回空記憶

Step 3: 意圖分類 (兩階段)

  • 階段 1 - 快速路徑:
    • 關鍵詞匹配 + 正則表達式
    • 計算置信度分數
    • 如果分數 ≥ 0.75，直接返回
  • 階段 2 - LLM 回退:
    • 置信度 < 0.75 或快速路徑失敗
    • 調用 LLM 進行智能分類
    • 約占 20% 流量
  • 意圖類型: ORDER, STORE, STOCK, CARGO, RETURN, GENERAL, COMPLAINT

Step 3.5: 生成立即確認 (NEW)

  • 根據 Step 3 的意圖類型生成快速確認
  • 確認示例:
    • ORDER: "正在查詢您的訂單..."
    • STORE: "為您搜尋最近的門店..."
    • COMPLAINT: "收到您的反饋，立即為您轉接..."
  • 目的: 提高用戶感知響應速度
  • 返回給 Make.com 作為 ack_text

Step 4: 並行數據抓取 (NEW)

  • 使用 asyncio.gather 同步執行多個數據源
  • 根據意圖決定抓取內容:
    • ORDER: Shopline + WMS + Fulfillment API 並行
    • STORE: 門店 API + 庫存查詢並行
    • STOCK: WMS API 查詢
    • CARGO: 物流系統查詢
  • 錯誤隔離: 一個源故障不影響其他源
  • 超時: 各源 3 秒，總超時 5 秒

Step 5: 組裝提示詞 + LLM 生成 (NEW)

  • 組裝項目: 用戶消息 + 記憶 + 上下文 + cs-persona
  • 調用 LLM (雙提供商):
    • 主提供商: OpenAI (GPT-4o / GPT-4o mini)
    • 回退提供商: Anthropic Claude
    • 控制: LLM_PROVIDER_ORDER 環境變數
  • 超時/錯誤 → 自動切換提供商
  • 兩個都失敗 → 降級回覆 (prewritten fallback)

Step 6: 保存記憶

  • 更新內容: 消息記錄、意圖、交互時間戳
  • 存儲位置: MemoryStore (開發) / D1 (生產)
  • TTL: 365 天

Step 7: 轉接檢查

  • 檢測 4 種轉接信號 (詳見 Section 6)
  • 如果檢測到信號 → 創建轉接工單
  • 發送通知至 cs-handoff 模組

  • 觸發條件:
    • 快速路徑置信度 < 0.75

  • LLM 配置:
    • 模型: GPT-4o mini (成本優化)
    • 溫度: 0.3 (確定性)
    • 超時: 2 秒
  • 失敗處理:
    • LLM 超時/錯誤 → 使用快速路徑最後已知結果
    • 兩者都失敗 → 默認為 GENERAL 意圖

def call_llm_with_fallback(prompt: str, intent: str):
    providers = env("LLM_PROVIDER_ORDER").split(",")

    for provider in providers:
        try:
            if provider == "openai":
                response = openai.ChatCompletion.create(
                    model="gpt-4o-mini",
                    messages=[{"role": "user", "content": prompt}],
                    temperature=0.3,
                    timeout=5
                )
                return response.choices[0].message.content

            elif provider == "anthropic":
                response = anthropic.messages.create(
                    model="claude-sonnet-4-20250514",
                    max_tokens=1024,
                    messages=[{"role": "user", "content": prompt}],
                    temperature=0.3
                )
                return response.content[0].text

        except (Timeout, APIError) as e:
            logger.warning(f"{provider} failed: {e}")
            continue

        except Exception as e:
            logger.error(f"{provider} critical error: {e}")
            continue

    # 所有提供商都失敗
    logger.critical("All LLM providers failed")
    return get_fallback_response(intent)

5. 並行數據抓取 (NEW)

使用 asyncio.gather 實現高效的多源並發

架構圖

實現細節

import asyncio
from typing import Dict, Any

async def fetch_order_context(order_id: str) -> Dict[str, Any]:
    """並行抓取訂單相關數據"""

    async def fetch_shopline():
        try:
            return await shopline_client.get_order(order_id)
        except Exception as e:
            logger.error(f"Shopline fetch failed: {e}")
            return {"error": str(e), "status": "failed"}

    async def fetch_wms():
        try:
            return await wms_client.get_fulfillment_status(order_id)
        except Exception as e:
            logger.error(f"WMS fetch failed: {e}")
            return {"error": str(e), "status": "failed"}

    async def fetch_cargo():
        try:
            return await logistics_client.get_tracking(order_id)
        except Exception as e:
            logger.error(f"Logistics fetch failed: {e}")
            return {"error": str(e), "status": "failed"}

    # 並行執行所有任務
    results = await asyncio.gather(
        fetch_shopline(),
        fetch_wms(),
        fetch_cargo(),
        return_exceptions=True  # 一個失敗不影響其他
    )

    shopline_data, wms_data, logistics_data = results

    return {
        "shopline": shopline_data,
        "wms": wms_data,
        "logistics": logistics_data,
        "timestamp": datetime.now().isoformat()
    }

錯誤處理策略

  • 隔離失敗: 一個數據源失敗不阻止其他源
  • 降級響應: 缺失數據生成通用回覆
  • 重試邏輯:
    • 暫時故障 (超時): 1 次重試
    • 永久故障 (4xx): 不重試
    • 伺服器故障 (5xx): 最多 3 次重試 (指數退避)
  • 日誌記錄:
    • 每個源的成功/失敗
    • 延遲測量
    • 錯誤詳情

性能最佳化

  • 連接池: 每個 API 客戶端維護 HTTP 連接池 (10-50 個連接)
  • 緩存: 5 分鐘內相同訂單 ID 的結果快取
  • 批量請求: Shopline API 支援批量訂單查詢
  • 定時清理: 過期快取每 1 小時清一次

  4. 離線隊列: 若 HQ 也滿，創建待機轉接工單，待首個顧問可用時立即分配
