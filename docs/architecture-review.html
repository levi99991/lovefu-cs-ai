一、邏輯問題檢視

🚨 CRITICAL cs-guard 的位置在生命週期圖上被誤導

架構圖把 Guard 標為「第 6 步」，會讓人誤以為 Guard 只在 LLM 前跑一次。但實際上 Guard 必須包住每一次 Shopline / WMS 呼叫，否則資料層的任何 skill 都能繞過安全檢查。

修正：把 Guard 畫成「橫跨整個資料層的外殼」，cs-shopline / cs-logistics 的每個 API 呼叫內部都必須 wms_safe_get() / shopline_safe_get()，不能繞過。

🚨 CRITICAL Intent 分類錯誤會汙染下游

如果 intent 判斷錯（例如顧客說「我床墊有破洞」被分到 CHAT），下游不會抓 WMS 也不會帶 knowledge，LLM 回答會失準。目前沒有「信心閾值 fallback」。

修正：加入 intent_confidence < 0.6 時走「多分類並行」— 同時抓 knowledge + shopline，讓 LLM 最後決定該用哪個。

⚠️ WARN 同一顧客連續多則訊息的競態

顧客可能連打三條訊息（「我想退貨」→「訂單 2024xxx」→「明天有空嗎」），目前每則獨立處理，Memory 寫回會有 race condition，後一則可能覆蓋前一則 profile。

修正：導入 message debounce（2 秒） — 同一 line_uid 的訊息進佇列，2 秒內新訊息併一起處理；或用 Redis lock 保序。

⚠️ WARN LLM_MODE=mock 誤開到 production 的風險

mock 模式的模板回覆拿來對真顧客講話會很尷尬。環境變數無守門。

修正：在 main.py 啟動時檢查 if ENV=="production" and LLM_MODE=="mock": raise RuntimeError。

⚠️ WARN 追客 5 則硬帽 vs 顧客主動回訊

目前的 14 日 5 則是「AI 主動發」的上限，但如果顧客主動回了 AI，是否算一則？邏輯沒界定。若算，顧客越回越多 AI 越啞。若不算，有漏洞。

修正：分開計數 — outbound_count（AI 主動）硬帽 5 則；顧客主動回訊走正常對話，不佔額度。

✓ OK GET-only 架構強制守門正確

單元測試 test_no_wms_safe_post_exists 已經在，從語言層面封死寫入路徑，這是對的。

三、實體店輔睡員銜接斷點

🚨 目前只有「追客」沒有「即時轉人工」

cs-instore 負責的是事後追客（S0/S1/S2/S3/S4），但沒有設計「AI 即時把這通對話 handoff 給實體店輔睡員」的斷點邏輯。這是最大的架構缺口。

四大轉人工觸發點

① 顧客明說
  • 「我要真人」
  • 「找客服」
  • 「轉人工」
  • 「我要找店員」
② AI 信心不足
  • intent 信心 < 0.5
  • 知識庫無匹配
  • 連續 2 次澄清失敗
③ 情緒紅線
  • 偵測到生氣 / 不耐
  • 重複問同一事 3+ 次
  • 投訴 / 客訴關鍵字
④ 高價值事件
  • 明確要預約到店
  • 準備下單但猶豫
  • 退換貨 / 安心睡啟動

Handoff 流程（即時斷點）

1

<100ms

銜接資訊包內容（給實體店輔睡員）

📋 必備欄位
  • 顧客 LINE 暱稱 + 綁定手機（若已綁）
  • 本次對話摘要（AI 自動產）
  • customer_journey（試用過哪些產品）
  • 最近一次訂單狀態
  • 會員等級 / 點數
  • 轉人工的觸發原因
💡 AI 建議話術
  • 顧客當下問題的建議回覆
  • 相關產品建議（knowledge 提供）
  • 可用優惠 / 活動
  • 該注意的禁忌或客訴歷史
  • 體驗店預約建議時段

銜接斷點的失敗模式

⚠️ 輔睡員不在線 / 下班時段

如果深夜或門市打烊時觸發 handoff，沒人接 → 顧客等待會更差。

修正：Handoff 前先檢查門市班表，若離線則 AI 自己撐場面 + 留言排明早處理 + 告知顧客：「現在輔睡員下班了，我幫你記下來，明天 10 點他一上班就回你 📱」

⚠️ AI 與真人回答衝突

AI 說可以退，店員說不行 → 品牌信任崩盤。

修正：退換貨 / 安心睡 / 價格議題一律不 AI 回答實質條件，只引導「我幫你轉給 xx 店的 Kevin 確認給你最準確的答案喔」。

⚠️ 顧客不知道自己在跟 AI 講話

AI 太像真人 → 接手後顧客發現是兩個人 → 尷尬。

修正：輔睡員身份統一稱「小島」，AI 講的事輔睡員會接續；如果要明確區分，在 handoff 時主動說「接下來由真人小島接手喔」。

四、Shopline / WMS API 串接風險

Shopline Admin API

問題嚴重度對應方案
Rate Limit
每分鐘請求上限		 高 全局 RateLimiter（token bucket），尖峰時把 cache TTL 自動拉長；429 錯誤自動退避 backoff
Token 過期 / 失效 高 啟動檢查 + 每小時健康檢查；失效時 AI 退回「我暫時查不到訂單唷，我先幫你聯絡店員」
Cache 導致舊資料
顧客剛下單但 AI 查不到		 中 顧客訊息若含「剛剛下單」「今天買」等時效關鍵字，強制 cache bypass；會員資料訂閱 Webhook 主動 invalidate
欄位變更 / 破版 中 schema validation；非預期欄位 → log warning + fallback「資料取得有點異常」

WMS 暢流 API

JWT 有效期管理 高 token 5 分鐘前自動 refresh；失效 retry 1 次 + 不成功立即告警
AES key 外洩 高 key 只存 Railway env var，不進 git；規劃 90 天 rotation 流程；審計日誌監控異常查詢頻率
PII 欄位位置漂移
上游改 schema 沒通知		 高 每個欄位用 allowlist 白名單遮罩，未知欄位一律當 PII 遮掉；每天自動 diff schema
跨系統不一致
Shopline 有 / WMS 沒（剛出單）		 中 先 Shopline 再 WMS；WMS 無資料時明確回「訂單還在準備出貨，明天再查喔」而非「查無訂單」
沒有 Webhook 反向推送
只能顧客主動問，AI 不知狀態變化		 中 重要狀態（出貨、到貨）改由 Shopline Webhook 通知，不依賴 WMS 主動；或 5 分鐘 polling 熱門訂單
