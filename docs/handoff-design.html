Signal Detection · State Machine · Advisor Reminder
v1.0 · 2026-04-15 · 已完工並通過 68 項測試

一、這個系統補什麼缺口

過去的缺口
  • cs-instore 只管「事後追客」（T+24h / 3d / 7d / 14d）
  • 沒有即時轉人工的斷點
  • AI 信心低或情緒紅線時，沒機制叫人來接
  • 輔睡員不在線，顧客沒人理也沒人知道
cs-handoff 補上的能力
  • ✓ 四大訊號即時偵測（顯性 / 信心 / 情緒 / 高價值）
  • ✓ 完整狀態機（pending → ack → resolved / missed）
  • ✓ 多通道通知（LINE Notify / Slack / Omnichat）
  • ✓ 五段式升級（5 / 15 / 30 / 60 分鐘）
  • ✓ 顧客端耐心話術自動觸發
  • ✓ 值班班表路由 + 離線 fallback

二、四大觸發訊號（signal_detector）

① EXPLICIT｜顧客明說 P0

  • 「我要真人」「找店員」「轉人工」
  • 「不要 AI」「你是機器人嗎」
  • 「找主管」「找經理」

② EMOTION｜情緒紅線 P0

  • 怒氣：「太扯」「受不了」「爛透」
  • 投訴：「消保官」「消基會」「告你」
  • 累積：連續 2 次不滿意、重複問 3+ 次

③ HIGH_VALUE｜高價值事件 P0

  • 退換貨 / 安心睡 / 瑕疵
  • 議價 / VIP 價 / 折扣談判
  • 近期到店（今天 / 明天 / 馬上）

③' HIGH_VALUE｜成交訊號 P1

  • 「預約」「想試躺」「想體驗」
  • 「怎麼下單」「要買」「分期」
  • 對話超過 6 輪未收斂

④ LOW_CONF｜AI 信心不足 P1

  • 顧客：「你沒聽懂」「我不是這個意思」
  • 連續 2 次澄清失敗
  • 意圖信心 < 0.35

P2 一般觀察 P2

  • 對話輪數累積（> 6 輪未解決）
  • 意圖信心介於 0.35~0.5
  • 非即時，Omnichat 旗標即可
優先順序：顯性 > 情緒 > 高價值 > 信心。第一個命中的訊號決定 priority，後續訊號可 升級（但不降級）。

四、五段式提醒機制（advisor_reminder）

T+0
首次派發
依 priority 決定通道：P0 全通道，P1 LINE + Omnichat，P2 僅 Omnichat 旗標
T+5 分
首席未接手 → 升級副手
首席 + 副手雙推播；priority 自動升 P0
T+15 分
全店廣播 + 店長 Email
所有 on-duty 輔睡員收到；同時顧客收到耐心話術 #1
📨 給顧客：「抱歉讓你久等了，輔睡員正在處理中，再稍等一下下唷 🌙」
T+30 分
HQ 總部升級
門市仍無回應 → 推送到 HQ 值班 JANE；顧客收到升級話術
📨 給顧客：「真的很不好意思，我請我們店長來協助，他會在 5 分鐘內過來喔！」
T+60 分
標記 MISSED + AI 解靜默
視為本批超時；寫入晨會檢討清單；AI 接手並告知顧客隔日回覆
📨 給顧客：「真的很抱歉今天輔睡員忙不過來，我已經幫你留訊息，明天一早他們就會回覆你。如果很急，可以直接打店裡電話：{store_phone}」
關鍵設計：提醒 timer 使用 threading.Timer（生產可替換為 Celery / Redis Queue），接手後立即 cancel_reminders()，避免空推播。

路由優先順序

1
顧客綁定的門市（memory.profile.preferred_store_id）+ 該店有開 + 有值班 → 送該店
2
其他門市有開 → 送第一家有值班的門市（可擴充：依顧客位置就近）
3
所有門市都關 + HQ 有開 → 送 HQ 值班
4
全離線 → 標記 offline，Omnichat 旗標 + 排明日營業時段 next_open_datetime()

七、與 cs-brain 的整合點

1
cs-brain /chat 完成 LLM 回覆後，呼叫 check_auto_handoff(line_uid, message, intent, memory)
2
signal_detector 按優先級掃四類訊號，命中 → 回 (True, reason)
3
handoff_manager.trigger() 建 handoff，advisor_roster 路由，notification_dispatcher.dispatch() 發通知
4
cs-brain 把 need_human=True + human_reason 寫回 ChatResponse，Make.com 看到就改發 handoff 訊息給顧客
5
advisor_reminder 排 5 / 15 / 30 / 60 分鐘 timer，接手後 cancel_reminders()
6
輔睡員按 Omnichat「我接手」→ 呼叫 POST /handoffs/{id}/acknowledge → 觸發 mark_agent_takeover()，AI 靜默 24 小時
7
對話結案 → POST /handoffs/{id}/resolve with outcome=resolved/booked/purchased

八、對輔睡員的 API（Omnichat 後台串接）

GET /handoffs/pending?store_id=taichung_7
列出該門市當前待接手的 handoff（依 priority 排序）
GET /handoffs/{handoff_id}
取單一 handoff 完整內容（摘要、建議話術、customer_journey）
POST /handoffs/{handoff_id}/acknowledge
輔睡員按「我接手」→ AI 靜默 24h，取消所有提醒 timer
Body: {"advisor_id": "U_advisor_kevin"}
POST /handoffs/{handoff_id}/resolve
結案並標記 outcome（成交 / 預約 / 無回應 / 取消）
Body: {"outcome": "booked", "note": "明天 15:00 體驗"}
GET /handoffs/missed/recent?hours=24
列出最近 24 小時超時未接的 handoff（晨會檢討用）

九、同場加映：本輪邏輯修正

問題解法檔案
Intent 分類慢（每次 call LLM） 兩段式：fast-path keyword / regex → 信心 ≥ 0.75 直接返回；低信心才 fallback LLM。預估省 -1.2 秒 intent_classifier.py
Intent 低信心會答非所問 LLM 回傳不合法時，退回 fast-path 結果（比純 CHAT 安全） intent_classifier.py
同顧客連發訊息 race condition 1.5 秒 debounce；第一則 handler 等窗口後合併處理，後續 silent=True app.py
LLM_MODE=mock 誤上 prod 啟動守門：ENV=production && LLM_MODE=mock → RuntimeError app.py _startup_safety_check
LLM 回覆後未檢查轉人工 Step 7 新增 check_auto_handoff()，need_human + handoff_reason 自動填回 app.py
cs-guard 位置誤導 確認 cs-shopline / cs-logistics 的每個呼叫都包在 safe_get 內（單元測試已守門） 架構文件更新
